Section 1 - What is web security?

Video on Introduction to Web Security

Web security is the act of protecting websites from attacks such as unauthorized access, modification, disruption, or destruction of data. On a number of occasions, websites will go down because of these attacks, and in some cases confidential user data is leaked to the public, exposing these users to risks.

Good website security requires an effort throughout the entire website: whether it is the web application itself, the configuration of the web server, or front-facing code. Even something like policies for confidential passwords creation or renewal plays a huge role in effective website security.

There are many ways to enable effective web security. Usually, if one is using a server-side web framework, effective defense mechanisms against common attacks are enabled. Configuring a web server by enabling HTTPS, for example, can help mitigate any other attacks that might occur.

Section 2 - Common Website Threats

This section will outline some of the most common threats to web security.

2.1 - Cross-Site Scripting (XSS)

Video on Cross-Site Scripting (XSS)

XSS is a category of attacks that allow an attacker to put client-side scripts through the websites into the browsers of other users. Since this code travels from the website itself into the browser, the code is usually trusted and has the ability to do a variety of things. For example, it can send a user’s authorization cookie to the attacker, which allows them to log into another website as if they were the user themselves. This gives the attacker the ability to do anything that the user would be able to do, such as access their bank account, their email, and other things.

XSS vulnerabilities are the most common security threat.

XSS vulnerabilities are reflected and persistent, depending on how the site returns the code to the browser.

Reflected XSS vulnerability occurs when user content to the server is returned immediately and unmodified in the browser.

For example, an attacker can construct a search link that contains malicious script as a parameter and email it to a user. If the user clicks this link, the script will be executed as the page loads. This would then give the attacker all the information they need to impersonate the user, being able to access any confidential or sensitive data belonging to them.

Persistent XSS vulnerability occurs when the malicious script is stored on the website and then redisplayed later, seemingly unmodified, for other users to execute.

For example, a discussion board that allows for unmodified HTML in the comment section could store a malicious script. When the comment is displayed, the script is executed and can send the users’ information to the attacker.

This attack is popular and powerful, since the attacker might not even engage directly with any of the users they are targeting.

Most common vulnerable data: Data from POST or GET requests is most common for XSS vulnerabilities. However, any data from a browser is vulnerable, whether it’s uploaded user files or cookie data.

Defense against XSS vulnerabilities:

Remove or disable any markup that could contain instructions to run code - in HTML, this includes elements such as script, object, embed and link .

Modifying user data so it can’t be used to run scripts is known as input sanitization. Many web frameworks sanitize user input by default.

2.2 - SQL Injection

Video on SQL Injection

SQL injection vulnerabilities give attackers the ability to execute SQL code on a database, allowing data access, modification, or deletion regardless of the attacker’s permissions. Successful attacks might utilize a hoax identity, create new identities with administration rights, access all the data on the server, or modify the data to make it unusable.

There are many different types of SQL injections: error based, boolean errors, time-based

How they occur: SQL injection vulnerabilities become present if user input that is passed into a query changes the meaning of the statement. To avoid this sort of attack, it needs to be ensured that user data passed into the query can’t change the nature of it.

One way to do this is to escape the characters in user input that have a special meaning in SQL. Web frameworks often just take care of character escaping.

2.3 - Cross-Site Request Forgery (CSRF)

Video on Cross-Site Request Forgery (CSRF)

CSRF attacks allow a malicious attacker to execute actions using credentials of a user without the user’s knowledge or consent.

For example, if an attacker knows a site that allows logged-in users to send money to other specific accounts by using an HTTP POST request, they can construct a form that includes their bank details and an amount of money as hidden details to email to other users with a submit button. If the user clicks the submit button, a POST request will be sent containing the transaction details and any cookies — the server checks the cookies, and if the user is logged in to the site, the transaction will be completed.

Prevention: A way to prevent this attack is for the server to require that POST requests include a user-specific site-generated secret. This secret would be supplied by the server when sending the web form - therefore the attacker would not be able to create their own form.

Note: Modern web frameworks usually include CSRF prevention mechanisms.

2.4 - Other Attacks

This section will overview some other attacks and threats to web security.

Video on Various Other Attacks

Clickjacking: An attacker routes clicks meant for a top level site to a hidden page underneath. For example, site might look like a legitimate bank site but only looks like it; captures login information that the attacker can use

Denial of Service (DoS): DoS is achieved by flooding a site with fake requests so that legitimate users can’t access the site. There might just be numerous requests or might consume large amounts of resources. Defenses usually identify and block malicious traffic while allowing legitimate messages through.

Directory Traversal: Attacker tries to access parts of the server file system that they shouldn’t be able to access - occurs when the user is able to pass filenames that include file system navigation characters. A solution to protecting against this type of attack is to sanitize the input.

File Inclusion: Attacker is able to specify an “unintended” file for display and when file is loaded, it might be executed on the web server or client side. A solution to protecting against this type of attack is to sanitize the input.

Command injection: Allows attacker to execute arbitrary system commands on the host operating system. it might be executed on the web server or client side. A solution to protecting against this type of attack is to sanitize the input.

General Points: Most of these attacks are successful when a web application trusts the data coming from a browser. So, the best way to protect yourself from a variety of attacks is to sanitize all user-originating data before it’s displayed in the browser, used in SQL queries, or passed to an operating system or file system call.

Section 3 - Mozilla Web Development Network’s Recommended Concrete Steps

Video on Recommended Steps / Conclusion

To outline what was presented in the video, here are a few key points to keep in mind in order to identify and patch some vulnerabilities to your website and web services:

  1. Use more effective password management - encourage strong passwords and the practice of changing it regularly, two factor authentication.

  2. Configure web server to use HTTPS and HSTS. HTTPS encrypts data sent between client and server - ensures that credentials, cookies, POST requests data and header information are not easily available to attackers.

  3. Keep track of the most popular threats (OWASP list), and address most common vulnerabilities first.

  4. Use vulnerability scanning tools to try and pinpoint the weaknesses of yuor system.

  5. Only store and display data that you need. For example, only show the last 4 digits of a credit card number, enough for the user to recognize stored information.
